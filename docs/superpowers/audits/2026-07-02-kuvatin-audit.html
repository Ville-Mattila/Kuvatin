The hardening plan below was executed in full and shipped as v2.0.1 (2026-07-03): the Critical, all 11 High and nearly every Medium/Low finding are fixed; the few deliberate exceptions (gui.rs split, ICC handling, Authenticode signing, Phase 7 features) are tracked in the plan. Health scorecard

The codebase is in good shape for its age — carefully commented, defensively written in its arithmetic, with real regression tests around the hardest-won GStreamer behavior. The dominant systemic weaknesses: errors are invisible in the shipped windowed build (every failure path reports to a stderr that doesn't exist), nothing added can be removed (clips, bin items), and third-party trust boundaries (WebP encoder, CI downloads) are unguarded.

AreaAssessment
kuvatin-core (images)C+Happy path solid and tested; but a panicking WebP encode on big images, a naming race under the parallel executor, and silently-dropped EXIF orientation/metadata undermine its core promise.
kuvatin-video (engine)B−Hard bugs already fought and regression-tested; lifecycle asymmetry (a failed export start strands the pipeline; nothing is removable/persistable) and an environment-dependent export fix are the weak spots.
kuvatin (GUI / shell)CCareful wiring, no reachable borrow-panics found — but error invisibility is systemic, both blocking modals are uncancellable on a frameless window, and table-stakes editor operations are missing.
Installer / CI / siteD+Ships working MSIs, but embeds unverified third-party binaries, runs zero tests before releasing, mixes per-machine install with per-user registration, and pushes ~100 GStreamer DLLs onto the system PATH.

Critical

1 finding — fix first
§1.1
C1guicoreA corrupt presets.toml permanently bricks startup — silently

Preset saves are non-atomic (fs::write); a crash mid-save leaves truncated TOML. Startup then fails at load_or_init(...)? and, with windows_subsystem="windows", the error goes nowhere: double-clicking the app simply does nothing, forever, until the user deletes an AppData file they've never heard of. Any preset missing one field (e.g. written by a newer version) triggers the same all-or-nothing parse failure.

Fix: atomic save (tmp + rename); on parse failure back up the bad file and fall back to built-ins instead of aborting; message-box any fatal startup error.
kuvatin-core/src/preset.rs:64–86 · kuvatin/src/gui.rs:124 · kuvatin/src/main.rs:1

High

11 findings
§2.1 §2.2
H1guiExport failures are invisible to the user

On a failed render the code writes a status string and closes the modal in the same tick — the message is never rendered; the real GStreamer error goes to eprintln!, which doesn't exist in the release build. The user watches the progress dialog vanish and later finds a broken partial file (this exact path shipped the 111 KB-file mystery). "Could not start export" has the same never-shown bug.

Fix: persistent error dialog carrying the actual pipeline message; delete the partial file.
kuvatin/src/gui.rs:1130, 1167–1175 · ui/app.slint:1826–1855
§2.2
H2guiExport cannot be cancelled; a hung render soft-locks the whole app

The export modal has no Cancel and its full-window TouchArea swallows every click — including the custom title-bar close button (the window is frameless). If the pipeline stalls without posting EOS/Error (a real failure mode), the only escape is Task Manager. The import modal has the identical no-escape design.

Fix: Cancel buttons (render teardown + partial-file cleanup), stall watchdog, and don't let overlays cover the window controls.
ui/app.slint:1826–1858, 1720–1753 · gui.rs:1140–1181
§2.4
H3videoguiNothing can be deleted — not a timeline clip, not a media-bin entry

Three independent reviewers confirmed: the engine has no remove_clip API at all, and the GUI's only removal anywhere is preset deletion. A wrong clip on the timeline is permanent until app restart; the clips map and GES layers grow monotonically for the process lifetime.

Fix: Project::remove_clip + trailing-layer pruning; ×-affordances and Delete key; re-key selection by id (indices will desync once removal exists).
kuvatin-video/src/project.rs:342, 458–852 · gui.rs · app.slint:464–525
§1.5
H4videoA failed export start leaves the preview permanently black

begin_render detaches the preview sink before its fallible steps; on any error it returns with the pipeline at Null, sinkless, possibly in RENDER mode — and the GUI's failure path never restores it. Preview and playback are dead for the rest of the session.

Fix: restore sink + FULL_PREVIEW on every error path inside begin_render.
kuvatin-video/src/project.rs:794–811 · gui.rs:1119–1131
§1.2
H5coreWebP encode panics on legitimate inputs

The webp crate's encode() unwraps internally; libwebp rejects dimensions >16383 px and out-of-range quality. Converting a phone panorama to WebP panics — and via the CLI/context-menu path, a hand-edited quality = 150 in presets.toml reaches the encoder unclamped.

Fix: encode_simple with mapped errors; clamp quality at the core boundary.
kuvatin-core/src/pipeline.rs:89 (+149) · quickrun.rs:12
§1.3
H6coreParallel batches can silently overwrite each other's outputs

ensure_unique is exists()-then-write with nothing reserving the name in between: photo.png and photo.jpg converting to WebP in the same folder race to photo-kuvatin.webp — one result is lost while both report success.

Fix: reserve with create_new(true) in a suffix-bumping loop.
kuvatin-core/src/naming.rs:68–82 · pipeline.rs:197,207
§4.1
H7coreEXIF orientation dropped — phone photos convert rotated

image::open doesn't apply the orientation tag and every encode path strips metadata: a portrait phone JPEG converts to a WebP/PNG lying on its side, with no warning. For a photo tool, this is the biggest everyday quality bug.

Fix: read orientation via ImageReader and apply_orientation before processing.
kuvatin-core/src/pipeline.rs:191, 219
§3.1
H8videoThe NVENC export fix silently deactivates on configured machines

The critical mfaacenc:0 derank is applied via GST_PLUGIN_FEATURE_RANK — but only if that env var is unset. Any user who has it set for other GStreamer software gets the documented 0-byte-export failure back, with zero diagnostics. The env write also happens from worker threads (set_var thread-safety hazard).

Fix: set ranks programmatically via the registry API after gst::init(); verify with the regression suite under a pre-set env var.
kuvatin-video/src/project.rs:207–224 · kuvatin/src/main.rs
§2.3
H9guiImport can hang forever, and failed files still enter the media bin

The import worker's synchronous discovery blocks indefinitely on e.g. an unreachable network path, behind an uncancellable modal. Discovery failures are ignored (let _ =) — the file is added to the bin anyway, producing dead entries that (see H3) can't be removed.

Fix: Cancel drains the queue; failed files excluded + one summary error.
gui.rs:206–215, 683–693 · app.slint:1720
§2.1
H10guiExplorer context-menu conversions fail silently

Registry commands hardcode preset names; rename or delete "Convert to WebP" in the GUI and the right-click action launches, errors to invisible stderr, and does nothing — unexplainable for the user.

Fix: native message box on quickrun failure in windowed builds; longer-term, stable preset IDs.
shell/windows.rs:16–21 · quickrun.rs:13–15
§5.1
H11shipCI bundles unverified downloads into the shipped installer

The release workflow downloads two GStreamer MSIs and the WiX zip with no checksum, executes them, and harvests those DLLs into the Kuvatin MSI — the strongest supply-chain lever into what every end user installs.

Fix: pin SHA-256 for all three; fail the build on mismatch.
.github/workflows/release.yml:27–32, 44–47
